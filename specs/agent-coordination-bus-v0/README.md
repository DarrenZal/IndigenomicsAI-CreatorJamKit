# Agent Coordination Bus — v0

A file-based JSONL bus implementing the 7 wire types from
[`specs/coordination-protocol-v0.md`](../coordination-protocol-v0.md).

This is one of the **meta-specs** for the jam: a substrate that
participating teams' agents can use to coordinate across teams without
silently sharing content, aggregating consent, or letting one team's
withdrawal be ignored.

## What it is

- Per-team append-only JSONL logs (`bus/teams/<team-id>.jsonl`)
- Append-only global log (`bus/global.jsonl`)
- A stdlib Python CLI: `scripts/jam/bus.py`
- Per-type validators rejecting refusal-boundary violations
- An audit subcommand verifying append-only invariant

## What it is not

- Not an HTTP service. v0 transport is filesystem; multi-team coordination
  happens by sharing the bus root (git, syncthing, shared volume) or by
  each team running their own bus and exchanging serialized messages.
- Not certification, authority, or grant of any permission. The bus
  records what was said. Humans + stewards remain the deciding voice.
- Not a trust score / reputation system. Every `share_request` is its
  own consent decision; the spec explicitly forbids reputation fields.

## Try it (60 seconds)

From the kit root:

```bash
# 1. Initialize a bus
python3 scripts/jam/bus.py init /tmp/my-bus --team team-kelp-bed-observers

# 2. Replay the worked demo
python3 scripts/jam/bus.py replay /tmp/coord-bus-demo \
  examples/agent-coordination-bus-demo/

# 3. See what happened
python3 scripts/jam/bus.py read /tmp/coord-bus-demo --global --summary

# 4. Verify append-only invariant
python3 scripts/jam/bus.py audit /tmp/coord-bus-demo

# 5. Validate a candidate message before posting
python3 scripts/jam/bus.py validate path/to/some-message.json
```

## Composition

- [`specs/coordination-protocol-v0.md`](../coordination-protocol-v0.md) — the wire-format contract (7 message types). This bus implements that contract.
- [`participant-agent-context/PROMPTS_FOR_AGENTS.md`](../../participant-agent-context/PROMPTS_FOR_AGENTS.md) — Prompt 4 (Collaboration Facilitator) is the natural caller of this bus from a participant's agent.
- [`tools/composition-merger.py`](../../tools/composition-merger.py) — `composition_propose` messages can be paired with this tool's `--dry-run` output for "expected_conflicts".
- [`tools/withdrawal-propagation.py`](../../tools/withdrawal-propagation.py) — `withdraw_notice` propagation manifests can be generated by this tool.
- [`specs/agentic-spec-drafting-loop-v0/`](../agentic-spec-drafting-loop-v0/) — emits `witness_observe` to this bus at the end of a run.

## Files

- `spec-v0.md` — the formal spec
- `../../scripts/jam/bus.py` — implementation
- `../../scripts/jam/tests/test_bus.py` — 20 unittest cases (all 7 types happy + 3 protocol failure cases + storage/audit)
- `../../examples/agent-coordination-bus-demo/` — worked example with 3 simulated teams

## Tests

```bash
python3 -m unittest scripts.jam.tests.test_bus -v
```

Expected: 20 tests OK (no external dependencies — stdlib only).

## Boundary

Validated lane = file-based coordination message exchange with refusal-
boundary enforcement. Not production agent transport, not authority,
not certification.
